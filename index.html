Core Modules
01 Auth Service

User signup, login, JWT access + refresh tokens, token revocation, session invalidation.

POST /auth/signup POST /auth/login POST /auth/refresh POST /auth/logout
02 Organization Service

Multi-tenancy core. Create orgs, invite members, assign RBAC roles (admin/member/viewer).

POST /orgs GET /orgs/:id POST /orgs/:id/invite POST /orgs/:id/roles
03 API Key Service

Generate and revoke API keys per org. Each key has scoped permissions, rate limits, and usage tracking.

POST /api-keys GET /api-keys DELETE /api-keys/:id GET /api-keys/:id/usage
04 Webhook System

Register endpoints, sign payloads with HMAC, deliver events via BullMQ with retry + dead-letter queue.

POST /webhooks/endpoints GET /webhooks/events POST /webhooks/test
05 Audit Logs

Immutable log of every action per org. Who did what, when, from which IP. Foundation for AI insights.

GET /audit-logs GET /audit-logs?userId= GET /audit-logs?action=
06 AI Usage Insights

Natural language queries over org API usage data. Ask "why did my usage spike?" and get plain-English answers.

POST /insights/query GET /insights/summary
Multi-Tenancy Model
Tenant A — Acme Corp
org_id: org_acme
users: alice, bob
api keys: key_abc, key_xyz
webhook: acme.com/hooks
Tenant B — BetaCo
org_id: org_beta
users: charlie, dan
api keys: key_123
webhook: betaco.io/events
Same DB · Same server · Isolated by org_id on every query · Cannot see each other's data
Webhook Delivery Flow
1
Event Triggered
org.member_added · user.created · api_key.generated
2
Event Persisted to DB
webhook_events table → status: pending
3
Job Pushed to BullMQ
Redis queue → webhook-delivery queue
4
Worker Delivers HTTP POST
HMAC-signed payload → customer's registered URL
5
Success or Retry
200 OK → mark delivered ✓  |  fail → retry x3 with backoff → dead-letter queue
